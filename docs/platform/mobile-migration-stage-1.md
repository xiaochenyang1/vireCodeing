# 移动端阶段 1 迁移策略

## 原则

- 保留现有本地认证流程作为测试基线。
- 新增 API adapter，不在页面中直接调用 `fetch`。
- 后端不可用时不破坏当前 401 个测试覆盖的本地 MVP 行为。

## 第一批新增文件

- `src/services/platformApiClient.ts`
- `src/services/platformAuthApi.ts`
- `src/services/platformRuntimeConfig.ts`
- `src/config/platformBuildConfig.ts`
- `scripts/write-platform-runtime-config.js`
- `__tests__/platformAuthApi.test.ts`

## 迁移步骤

1. 为认证 API 新增独立 adapter。
2. 用 Jest mock `fetch` 验证请求路径、请求体、鉴权头和响应映射。
3. adapter 已覆盖 `sendCode`、`login`、`refresh`、`logout` 和 `getMe`。
4. `AuthScreen` 已支持注入平台 adapter：注入后登录验证码发送和登录提交走平台 API，验证码本地过期判断会使用平台 `sendCode` 返回的 `expireSeconds`，并在认证页说明里标明登录已接入平台认证接口、注册仍保留本地演示；默认未注入时继续走本地固定验证码。
5. 根 `App` 已支持通过 `platformApiBaseUrl` 或 `globalThis.__TRUCK_PLATFORM_CONFIG__.apiBaseUrl` 创建平台认证 adapter 并注入 `AuthScreen`，默认未配置时继续走本地固定验证码。
6. 根 `App` 冷启动时会用已保存的 refreshToken 调用平台 `/auth/refresh`，成功后更新本地 token 快照。
7. `index.js` 已在 App 注册前安装 `src/config/platformBuildConfig.ts`，可通过 `TRUCK_PLATFORM_API_BASE_URL=... npm run platform:config:write` 在构建前生成平台 API baseUrl。
8. Android `preBuild` 已挂载 `writePlatformRuntimeConfig`，可读取 Gradle 属性 `truckPlatformApiBaseUrl` 或环境变量 `TRUCK_PLATFORM_API_BASE_URL` 自动生成移动端平台 API 配置。
9. iOS shared scheme 的 Build pre-action 已挂载同一生成脚本，`xcodebuild` 或 Xcode 构建前会在仓库根目录执行 `node scripts/write-platform-runtime-config.js`。
10. GitHub Actions 验证流水线已设置 `TRUCK_PLATFORM_API_BASE_URL` 并在根/API 检查前执行 `npm run platform:config:write`。

## 当前边界

- `AuthScreen` 默认仍使用本地固定验证码和本地会话；注入平台 adapter 时可调用后端验证码发送和登录接口，会用平台 `sendCode` 返回的 `expireSeconds` 更新本地验证码会话过期时间，并把 token 快照写入认证会话，同时页面顶部说明会从“不接后端”切换为“登录已接入平台认证接口，注册仍保留本地演示”；验证码发送上游失败时会把 `AUTH_CODE_DELIVERY_FAILED` 映射成明确的短信服务不可用提示，验证码发送限流会把 `AUTH_CODE_RATE_LIMITED` 映射成获取过于频繁提示，网络失败会把 `NETWORK_ERROR` 映射成网络不可用提示；根 `App` 已支持通过显式 `platformApiBaseUrl`、构建前生成的 `platformBuildConfig` 或全局运行时配置 `globalThis.__TRUCK_PLATFORM_CONFIG__.apiBaseUrl` 注入该 adapter，并在冷启动时用已保存的 refreshToken 调用 `/auth/refresh` 换取新的 token 快照，随后用 `/me` 回填当前用户手机号；启动期 refresh 或 `/me` 网络失败/临时接口异常会保留本地未过期会话，只有 `AUTH_REFRESH_TOKEN_INVALID`、`AUTH_ACCESS_TOKEN_INVALID` 或 `AUTH_USER_DISABLED` 这类明确认证失效/禁用错误会清理本地会话。
- `apps/api` 已提供阶段 1 认证接口边界；已有 PrismaAuthRepository、`PrismaService`/`PrismaModule`、AuthModule 注入链路和初始 Prisma migration SQL，可把用户和 refresh session 映射到 Prisma client，refresh token 是不包含用户 ID/TTL 的 opaque 随机值，会以 hash 存储，刷新时从 active session 读取用户 ID、撤销旧 refresh session 并保存新的 refresh token session。
- 后端实际挂载 Nest 全局前缀 `/api`，阶段 1 OpenAPI server 已对齐为 `http://localhost:3000/api`；移动端运行时 `platformApiBaseUrl` 会 trim 并去掉尾部 `/`，`platformApiClient` 自身也会规范化 baseUrl/path 拼接，所以 `http://localhost:3000/api/` 和 `http://localhost:3000/api` 会落到同一个接口前缀。
- 认证接口已校验手机号、验证码、用户类型、`deviceId` 和 `refreshToken` 的基础格式；`refreshToken` 请求边界当前只接受 `refresh.<UUID>`，会拒绝旧的 `refresh.local-user-...604800` 语义化 token。
- 移动端 `platformAuthApi.refresh()` 和 `platformAuthApi.logout()` 会在请求前裁剪 `refreshToken/deviceId` 两端空白，只接受后端当前 opaque token 形态 `refresh.<UUID>`；非法 token session 请求会抛出 `PLATFORM_AUTH_TOKEN_SESSION_REQUEST_INVALID`，不会把旧语义 refresh token、access token、空设备 ID 或非对象请求打到后端。
- `platformApiClient` 会把非 2xx 错误、2xx 业务错误、网络失败和无效成功 envelope 都映射成带 `code`、`status` 和 `requestId` 的错误对象；auth adapter 已区分公开认证入口和受保护接口，验证码、登录、注册、找回、refresh、logout 不携带 bearer，`/me` 和登录后改密才携带 access token；服务层受保护请求如果拿不到 `accessToken`，会在调用 `fetch` 前抛出 `AUTH_ACCESS_TOKEN_MISSING`/`status: 0`，不再允许无 bearer 请求漏到 `/me`、`/shipper/orders`、草稿、地址簿或常用路线这类受保护接口；设置页平台改密在调用 `/auth/change-password` 前也会检查本地 `accessToken`，缺 token 时提示重新登录且不发无 bearer 请求；网络失败固定为 `NETWORK_ERROR`/`status: 0`，无效响应固定为 `PLATFORM_RESPONSE_INVALID`，并可通过 `getRequestId` 配置发送 `x-request-id`，页面接入时应优先按稳定 `code` 映射用户提示，不要再靠字符串判断错误。
- 本地认证会话刷新时会区分平台 token 会话和本地演示会话：平台会话只保留原 accessToken/refreshToken/issuedAt/expiresAt 快照，不会延长服务端 token 到期时间；本地演示会话继续使用 7 天本地 TTL。平台 token 的真实续签只能走 `/auth/refresh`。
- 阶段 1 OpenAPI 已明确认证相关成功响应的 `OK` envelope，`sendCode`、`login`、`refresh`、`logout` 和 `/me` 都有对应 response schema，并已声明可选 `x-request-id` header；后端认证成功响应和业务错误响应都会把 `x-request-id` 透传到响应体 `requestId`，移动端 adapter 继续只向页面暴露解包后的 `data`。
- `/me` 已由后端阶段 1 `AccessTokenGuard` 保护，guard 只接受单个严格 `Bearer <token>` header，会拒绝空 token、多段 token、错误 scheme 和数组 header；access token 已升级为 HS256 JWT，校验签名和 `exp` 过期时间。
- 生产环境 JWT secret 已有基础防呆：拒绝开发占位符和少于 32 字符的密钥。
- 后端验证码策略已按环境切换：非生产保留固定 `devCode`，生产环境生成随机 6 位验证码且不回传 `devCode`；验证码发送记录已写入 `VerificationCode.codeHash`，验证码发送已接入可替换 sender，`parseEnv` 已覆盖 `SMS_PROVIDER`、`SMS_WEBHOOK_URL`、`SMS_WEBHOOK_TOKEN` 和 `SMS_WEBHOOK_TIMEOUT_MS` 启动前校验，生产环境缺少短信 provider、配置非 HTTPS webhook、webhook token 少于 16 字符或设置非法 timeout 都会拒绝启动，webhook sender 会使用 `SMS_WEBHOOK_TIMEOUT_MS` 控制请求超时且默认 5000ms，sender 失败会返回 `AUTH_CODE_DELIVERY_FAILED` 和 HTTP 502；服务端已加 60 秒重发冷却、同手机号同用途 1 小时 5 次发送上限、`VERIFICATION_CODE_TTL_SECONDS` 可配置验证码有效期且默认 300 秒、`AUTH_CODE_RATE_LIMITED` 限流错误码和 HTTP 429，以及 `AUTH_CODE_EXPIRED` 过期错误码；具体短信供应商模板/签名/回执/重试、Redis/IP/设备维度分布式风控和真实 PostgreSQL 环境验收仍未接入。
- refresh token 仍是 opaque token，并通过后端 refresh session hash 存储、轮换和撤销，不是 JWT；后端请求校验和 `TokenService` 内部计算都只接受 `refresh.<UUID>`，客户端应原样保存后端返回值，不应拼接或解析 token 内容。
- 本地 PostgreSQL Compose 入口、`db:dev:postgres:up/down`、`db:postgres:doctor` 和 `db:postgres:bootstrap` 已补；`bootstrap` 已串行覆盖迁移部署、阶段 1 seed、认证冒烟和订单创建/列表/详情冒烟；Android `preBuild`、iOS shared scheme 和 GitHub Actions 验证流水线已自动挂载 `platform:config:write` 的底层生成脚本；下一步应在可用 Docker CLI 或真实 PostgreSQL 环境里跑通连接/迁移/seed/auth-smoke/order-smoke，再补 KMS/密钥轮换、正式发布流水线和完整设备会话风控。

## 阶段 2 追加边界

- 货主订单第一片真实化已新增 `src/services/platformOrderApi.ts` 和 `src/services/platformOrderMapper.ts`，覆盖订单创建、列表、详情 adapter 和后端订单 DTO 到现有 `RecentOrder` 的映射。
- 移动端 `platformOrderApi` 的订单详情、编辑、取消、确认送达、状态推进、异常上报、改单申请和评价接口会先裁剪 `orderId` 两端空白，裁剪后为空或运行时非字符串会在请求前抛出 `PLATFORM_ORDER_ID_INVALID`，不会把空路径、`null`、数字或对象拼进订单 URL。
- 根 `App` 在配置 `platformApiBaseUrl` 且已有平台 access token 时，发布新订单会优先调用 `/shipper/orders`；成功后保存平台订单号和已同步状态，网络或平台接口失败时继续保存本地待同步订单，订单同步状态会保留 `operation: create`，同步状态和队列项会按当前业务时钟写入结构化 `updatedAtIso`；如果发布新订单时 adapter 已配置但本地 access token 已丢失，会保留本地订单并写入 `operation: create` 失败队列，提示重新登录后同步，不再退回普通本地 pending 状态；详情页订单同步队列点击重试会重新 `POST /shipper/orders` 创建平台订单，成功后用平台订单号、后端主键和已同步状态替换本地订单，失败则继续保留本地失败队列；如果平台 adapter 已配置但本地 access token 已丢失，重试会继续保留失败队列并提示需要重新登录，不会按本地演示模式直接清成已同步。
- 后端已新增 `PUT /shipper/orders/{orderId}`，只允许当前货主自己的 `waiting` 订单直接修改，非待接单状态返回 `409 ORDER_STATE_INVALID`；移动端编辑待接单平台订单时会调用 `platformOrderApi.updateOrder()`，成功后用后端订单回写本地，失败时保留本地修改并标记同步失败，详情页订单同步队列点击重试会重新 `PUT /shipper/orders/{orderId}` 更新原平台订单；如果发布编辑时 adapter 已配置但本地 access token 已丢失，会保留本地修改并写入 `operation: update` 失败队列，提示重新登录后同步，不再退回普通本地 pending 状态。
- 移动端 `platformOrderApi.createOrder()` 和 `platformOrderApi.updateOrder()` 会在请求前按后端发单 schema 归一化订单主体：必填字符串裁剪后不能为空，空白的可选裁剪字段不发送，备注/描述长度、图片数量、手机号、装货时间、布尔字段、`pricingMode/paymentMethod`、固定价金额、议价订单夹带金额、优惠券字段成套性和实付金额关系都会先校验；非法请求会抛出 `PLATFORM_ORDER_REQUEST_INVALID`，不会把脏发单或编辑请求打到后端。
- 后端已新增 `POST /shipper/orders/{orderId}/cancel`，会把当前货主自己的非完成/非已取消订单更新为 `cancelled` 并追加 `cancelled` 事件，非法状态返回 `409 ORDER_STATE_INVALID`；移动端平台订单详情页取消订单时会调用该接口，成功后用后端订单回写本地并保留本地取消原因卡，失败时保留本地取消记录并标记同步失败，详情页订单同步队列点击重试会重新 `POST /shipper/orders/{orderId}/cancel` 取消原平台订单；如果平台 adapter 已配置但本地 access token 已丢失，取消会保留本地取消记录并写入 `operation: cancel` 失败队列，提示重新登录后再同步。违约金计算、退款进度、客服审核和司机端同步仍未真实化。
- 移动端 `platformOrderApi.cancelOrder()` 会在请求前校验取消请求体本身必须是对象，`reasonText` 会裁剪两端空白且必须为 1 到 50 个字符，`description` 会裁剪两端空白、空白描述不发送且最多 200 个字符；非法请求会抛出 `PLATFORM_ORDER_CANCEL_REQUEST_INVALID`，不会把脏取消原因打到后端。
- 后端已新增 `POST /shipper/orders/{orderId}/status`，只允许当前货主自己的订单按 `waiting -> loading -> transporting -> confirming` 顺序推进，非法跳转返回 `409 ORDER_STATE_INVALID`；移动端平台订单详情页本地状态操作会调用 `platformOrderApi.advanceOrderStatus()`，成功后用后端订单回写本地并保留本地运行态记录卡，失败时保留本地状态变更并标记同步失败，详情页订单同步队列点击重试会重新 `POST /shipper/orders/{orderId}/status` 推进原平台订单。真实司机端接单、装货确认、运输轨迹、送达确认和状态冲突审计仍未真实化。
- 移动端 `platformOrderApi.advanceOrderStatus()` 会在请求前校验状态推进请求体本身必须是对象，`nextStatus` 只能是 `loading`、`transporting` 或 `confirming`；`null`、字符串请求体、空值或非推进目标状态都会抛出 `PLATFORM_ORDER_STATUS_REQUEST_INVALID`，不会把脏状态推进请求打到后端。
- 后端已新增 `POST /shipper/orders/{orderId}/complete`，只允许当前货主自己的 `confirming` 订单确认送达，成功后更新为 `completed` 并追加 `completed` 事件，非 `confirming` 状态返回 `409 ORDER_STATE_INVALID`；移动端平台订单详情页点击“确认送达”会调用该接口，成功后用后端完成态订单回写本地并保留本地运行态记录卡，失败时保留本地完成记录并标记同步失败，详情页订单同步队列点击重试会重新 `POST /shipper/orders/{orderId}/complete` 完成原平台订单。真实支付结算、托管放款、司机端送达确认、评价中心记录和完成后的资金/发票联动仍未真实化。
- 后端已新增 `POST /shipper/orders/{orderId}/exception`，只允许当前货主自己的 `transporting` 或 `confirming` 订单上报异常，成功后追加 `exception_reported` 事件且不改变订单状态；移动端平台订单详情页提交异常时会调用 `platformOrderApi.reportException()`，成功后用后端订单回写本地并保留本地异常记录卡，失败时保留本地异常记录并标记同步失败，详情页订单同步队列点击重试会重新 `POST /shipper/orders/{orderId}/exception` 上报原平台订单异常。真实异常图片上传、客服工单、客服处理记录和异常处理状态同步仍未真实化。
- 移动端 `platformOrderApi.reportException()` 会在请求前校验异常上报请求体本身必须是对象，`typeLabel` 会裁剪两端空白且必须为 1 到 30 个字符，`description` 会裁剪两端空白且必须为 6 到 200 个字符，`photoCount` 如传入必须是 0 到 6 的整数；非法请求会抛出 `PLATFORM_ORDER_EXCEPTION_REQUEST_INVALID`，不会把脏异常上报请求打到后端。
- 后端已新增 `POST /shipper/orders/{orderId}/change-request`，只允许当前货主自己的 `loading`、`transporting` 或 `confirming` 订单提交修改申请，成功后追加 `change_requested` 事件且不改变订单主体或状态；`waiting` 订单仍走 `PUT /shipper/orders/{orderId}` 直接编辑。移动端平台订单详情页提交修改申请时会调用 `platformOrderApi.submitChangeRequest()`，成功后用后端订单回写本地并保留本地修改申请卡，失败时保留本地修改申请并标记同步失败，详情页订单同步队列点击重试会重新 `POST /shipper/orders/{orderId}/change-request` 提交原平台订单修改申请。真实客服审核、费用重算、退款差额、司机确认和审核结果同步仍未真实化。
- 移动端 `platformOrderApi.submitChangeRequest()` 会在请求前校验修改申请请求体本身必须是对象，`description` 会裁剪两端空白且必须为 1 到 200 个字符；非法请求会抛出 `PLATFORM_ORDER_CHANGE_REQUEST_INVALID`，不会把脏修改申请请求打到后端。
