# 完整货运平台总体架构与阶段拆分设计

## 1. 背景与当前状态

当前仓库是 React Native 货主端本地 MVP 原型，不是完整货运平台。已有货主端本地登录、首启引导、首页、发单、订单列表、订单详情、消息中心、客服帮助、个人中心、草稿持久化、本地同步边界和 176 个 Jest 行为测试。

当前缺口集中在真实业务基础设施：

- 没有后端服务、数据库模型、真实 API client、真实 token/refresh token 机制。
- 没有司机端、后台管理端。
- 地图定位、地址搜索、路线规划、实时轨迹只是本地展示。
- 图片/证照/异常凭证没有真实上传和对象存储。
- 微信/支付宝支付、资金托管、退款、分账、提现、发票真实开具都未接入。
- 推送、IM、客服工单真实流转、运营配置、埋点和风控未开发。

完整平台建设不能继续把能力堆进单个 React Native 项目。后续必须拆成移动端、后端、后台、第三方服务接入和数据基础设施，按阶段交付。

## 2. 目标

把当前货主端本地 MVP 升级为可真实运行的货运平台，覆盖：

- 货主端：真实认证、发单、订单管理、支付、评价、发票、客服和消息。
- 司机端：认证、车辆资质、订单大厅、报价/接单、订单执行、收入和提现。
- 后台端：审核、订单、异常、支付退款、发票、优惠券、运营配置和风控管理。
- 公共能力：地图、定位、路线、上传、推送、IM、支付、结算、审计、埋点。
- 后端能力：认证鉴权、订单状态机、调度匹配、资金账务、消息通知、文件、审核和后台 API。

## 3. 非目标

第一轮完整平台规划不直接实现所有功能，也不把微服务、司机端、后台和支付一次性塞进当前 RN 工程。

短期不做：

- 大规模微服务拆分。
- 自研 IM 底层协议。
- 自研地图和支付通道。
- 完整财务自动对账系统。
- 多租户 SaaS 化。

这些能力可以作为中后期演进项，不能压到第一批落地里。否则项目会像铁锅炖自己，热闹但没法吃。

## 4. 建设路线选择

### 4.1 推荐路线：模块化单体后端 + 双端 RN + Web 后台

后端先做一个模块化单体服务，内部按认证、用户、订单、支付、消息、文件、客服、运营、后台拆模块。移动端保留当前 React Native 技术栈，逐步引入真实 API、导航、SDK 和缓存层。后台独立 Web 工程。

优点：

- 交付最快，最适合当前从本地 MVP 升级。
- 数据事务和订单状态机更容易保证一致。
- 运维复杂度低，团队规模要求低。
- 后期可以按边界拆服务。

缺点：

- 模块边界要管住，否则单体会变成大泥球。
- 高并发地图位置流、IM、推送等能力后期可能需要独立服务。

### 4.2 备选路线：微服务优先

认证、订单、支付、调度、消息、文件、后台各自服务化。

优点是边界清楚、可独立伸缩；缺点是初期成本高，需要服务治理、链路追踪、消息总线、分布式事务和更多运维能力。当前项目还没后端，直接上这套容易把地基修成机场。

### 4.3 备选路线：BaaS/第三方能力优先

短信、上传、推送、IM、地图、支付尽量接第三方，后端只做业务编排。

优点是快；缺点是核心业务规则、账务状态、订单状态机仍必须自控，供应商锁定也会更明显。

结论：第一阶段采用 4.1，第三方服务只承担地图、支付、短信、上传、推送和 IM 的底层能力。

## 5. 目标系统架构

### 5.1 移动端

- `shipper app`：当前 React Native 货主端继续演进。
- `driver app`：第一阶段和货主端共用 RN 工程，按 `userType` 进入司机模块；司机端业务目录和货主端隔离，后续用户量或发布节奏分化后再拆独立 RN App。
- 引入正式导航：React Navigation。
- 引入 API client：统一鉴权、错误处理、重试、分页、上传、刷新 token。
- 引入数据缓存层：服务端状态和本地草稿/离线队列分开管理。
- 引入 SDK 适配层：地图、定位、上传、支付、推送、IM。

### 5.2 后端

默认选择 Node/NestJS。当前前端是 TypeScript，NestJS 更容易共享 DTO、校验规则和 OpenAPI 生成链路，适合从当前 RN MVP 快速起后端地基。Java/Spring Boot 只作为团队已有成熟 Java 支付和运维体系时的替代方案，不作为本规划默认路线。

后端模块：

- Auth：验证码、密码、token、refresh token、设备会话、权限。
- User：货主/司机资料、实名、企业、车辆和资质认证。
- Order：订单创建、报价、接单、执行、取消、异常、评价、状态机。
- Dispatch：订单大厅、司机匹配、距离估算、报价排序。
- Map：地址解析、经纬度、路线距离、轨迹和位置上报。
- Payment：支付单、预冻结/托管、扣款、退款、分账、提现。
- Coupon/Invoice：优惠券、计价、核销、发票申请和发票状态。
- File：上传签名、文件记录、文件用途、审核资料。
- Message：站内信、推送、已读状态、模板通知。
- IM/Support：会话、客服工单、异常处理记录。
- Admin：审核、订单、资金、运营配置、风控、数据看板。
- Audit：关键操作审计、风控日志、状态变更日志。

### 5.3 数据基础设施

- 主库：PostgreSQL。
- 缓存：Redis，用于验证码、限流、token 黑名单、在线司机状态、临时位置。
- 对象存储：S3 兼容对象存储，用于证件、货物图片、异常凭证、回单、发票文件。
- 消息队列：支付回调、推送、IM 通知、订单状态异步事件。
- 日志与监控：接口日志、业务日志、错误日志、审计日志和告警。

## 6. 核心数据模型草案

第一批后端至少需要以下模型：

- `users`：用户基础信息、手机号、用户类型、状态。
- `auth_sessions`：设备、token 版本、登录时间、过期时间。
- `shipper_profiles`：货主资料、个人/企业状态。
- `driver_profiles`：司机资料、实名状态、服务城市、评分。
- `vehicles`：车辆类型、车长、载重、车牌、认证资料、审核状态。
- `certification_records`：实名、企业、车辆、资质审核记录。
- `addresses`：常用地址、联系人、经纬度、地址来源。
- `contacts`：常用联系人。
- `orders`：订单主表、状态、货主、司机、价格、支付状态。
- `order_cargo`：货物类型、重量、体积、数量、描述、凭证。
- `order_locations`：装货/卸货地址、联系人、经纬度、时间要求。
- `order_requirements`：车型、车长、尾板、篷布、增值服务。
- `driver_quotes`：司机报价、到达时间、备注、是否选中。
- `order_events`：状态流转、操作人、时间、原因。
- `order_exceptions`：异常类型、说明、凭证、客服处理状态。
- `payments`：支付单、渠道、金额、状态、回调流水。
- `settlements`：司机收入、平台服务费、分账、提现状态。
- `refunds`：退款金额、原因、状态、渠道流水。
- `coupons` / `coupon_usages`：优惠券发放、锁定、核销。
- `invoices`：发票申请、抬头、订单、金额、状态、文件。
- `files`：文件 URL、用途、上传人、审核关系。
- `messages`：站内消息、推送模板、已读状态。
- `chat_sessions` / `chat_messages`：货主司机或客服会话。
- `support_tickets`：投诉建议、异常工单、处理记录。
- `admin_users` / `roles` / `permissions`：后台账号和权限。
- `audit_logs`：关键操作日志。

## 8. 移动端改造策略

当前货主端本地 MVP 不能一次性删掉 mock。建议逐步迁移：

1. 保留本地状态作为离线兜底和失败队列。
2. 新增 API client，不直接在页面里 `fetch`。
3. 用 adapter 把后端 DTO 映射到现有 `RecentOrder` 等 UI 模型。
4. 一个模块一个模块替换 mock：认证、订单列表、订单详情、发单、个人中心。
5. 引入正式导航，替换顶层字符串切屏。
6. 把本地同步状态从演示文案改成真实失败队列和重放机制。

页面迁移优先级：

1. 认证和会话。
2. 订单列表/详情只读接口。
3. 发单创建订单。
4. 草稿、常用地址和联系人。
5. 个人中心认证、优惠券、消费、发票。
6. 地图、支付、上传、推送和 IM。

## 9. 司机端设计边界

司机端 MVP 优先覆盖真实接单闭环：

- 司机注册/登录。
- 实名认证、车辆认证、资质认证。
- 订单大厅：附近订单、筛选、距离、价格、装货时间。
- 报价或抢单：根据订单定价模式处理。
- 接单后详情：货物、路线、联系人、导航入口。
- 状态流转：到达装货点、装货完成、运输中、送达、上传回单。
- 位置上报：前台/后台定位策略、频率和电量控制。
- 收入概览：订单收入、待结算、可提现。

司机端不应复用货主端页面硬改。可复用底层：

- API client。
- 认证会话。
- 上传适配。
- 地图适配。
- 通用 UI 组件。
- 类型定义和 DTO mapper。

## 10. 后台管理端设计边界

后台第一批要服务平台运营，而不是做花里胡哨的数据大屏。

第一批后台模块：

- 登录和角色权限。
- 用户管理。
- 司机/车辆/企业/实名审核。
- 订单查询和状态查看。
- 异常/投诉工单处理。
- 支付、退款和分账查看。
- 发票审核和下载状态维护。
- 优惠券活动配置。
- 运营配置：城市、车型、价格规则、引导页和版本提示。
- 审计日志。

后台不能直接改数据库状态，所有操作必须走后端业务服务并写审计日志。

## 11. 地图、上传、支付、推送和 IM 接入边界

### 地图

- 移动端默认接高德地图 SDK。
- 后端保存经纬度、地理编码结果和路线距离快照。
- 司机位置写 Redis 热态，订单轨迹落库做历史回放。
- 位置上报频率按订单状态动态调整。

### 上传

- 后端签发上传凭证。
- 移动端直传对象存储。
- 上传完成后提交文件记录。
- 文件必须绑定用途：证件、货物、异常、回单、发票。

### 支付

- 支付单由后端创建。
- 移动端只调 SDK，不计算最终金额。
- 支付回调只信任服务端验签结果。
- 订单完成、取消、退款、分账都写资金流水。
- 所有金额使用整数分，避免浮点金额坑人。

### 推送

- 站内信和系统推送分开。
- 重要订单状态必须有站内信记录，系统推送只是触达通道。
- 用户设置控制推送类型，不影响站内消息生成。

### IM

- MVP 优先第三方 IM。
- 后端保存会话关系、订单关联和客服工单关联。
- 货主司机聊天必须受订单状态和隐私规则控制。

## 12. 分阶段实施计划

### 阶段 0：平台工程准备

交付物：

- 完整平台目录规划。
- 后端技术栈定稿。
- 数据库 ERD 初稿。
- OpenAPI 规范初稿。
- 当前货主端 mock 到真实 API 的迁移清单。

验收：

- 规格文档可支持后端、移动端和后台并行拆任务。
- 不修改现有货主端行为。
- 现有 `npm test`、`npx tsc --noEmit`、`npm run lint` 通过。

### 阶段 1：后端基础与认证

交付物：

- 后端工程。
- 数据库迁移。
- 用户、认证、验证码、token、refresh token。
- 文件上传签名基础。
- API client 接入移动端认证。

验收：

- 货主端能用真实登录进入首页。
- 本地会话替换为真实 token 会话。
- 验证码、限流、过期、刷新、退出登录有测试。

### 阶段 2：货主端订单真实化

交付物：

- 创建订单。
- 订单列表。
- 订单详情。
- 草稿同步。
- 常用地址/联系人。
- 订单状态事件。
- 异常、评价、取消、修改申请基础接口。

验收：

- 当前货主端主链路从 mock 切到后端。
- 本地失败队列能记录真实请求失败并可重试。
- 订单状态机由后端控制，前端不再随便改状态。

### 阶段 3：司机端 MVP

交付物：

- 司机认证入口。
- 车辆认证。
- 订单大厅。
- 报价/接单。
- 装货、运输、送达状态流转。
- 位置上报基础。
- 收入概览。

验收：

- 货主发布订单后司机端可看到、报价或接单。
- 司机状态流转能驱动货主端订单状态变化。
- 订单事件完整落库。

### 阶段 4：地图与调度

交付物：

- 定位权限。
- 地址搜索和地图选点。
- 路线距离估算。
- 司机实时位置。
- 轨迹回放。
- 订单大厅排序和基础匹配规则。

验收：

- 发单地址有经纬度。
- 订单距离由地图服务计算。
- 运输中订单可展示司机位置。
- 司机端位置上报不会压垮后端。

### 阶段 5：支付、结算、发票

交付物：

- 微信/支付宝支付单。
- 支付回调验签。
- 资金托管状态。
- 取消退款。
- 司机收入和提现。
- 发票真实申请和文件下载。

验收：

- 支付状态只能由后端回调和后台审核驱动。
- 金额、优惠、退款、分账有完整流水。
- 订单完成后能形成消费记录和可开票记录。

### 阶段 6：上传、推送、IM 和客服

交付物：

- 图片/证件/异常/回单上传。
- 站内信和系统推送。
- 货主司机 IM。
- 客服工单和处理记录。

验收：

- 所有凭证文件有真实 URL、用途和归属。
- 订单关键事件会生成站内信。
- 客服能处理异常和投诉。

### 阶段 7：后台、运营和风控

交付物：

- Web 后台。
- 审核、订单、支付退款、发票、优惠券、客服、运营配置。
- 风控黑名单、信用分、审计日志。
- 基础数据看板。

验收：

- 运营人员能完成审核和异常处理。
- 所有后台关键操作写审计日志。
- 风控规则能影响下单、接单和提现。

## 13. 测试策略

- 前端保留当前 Jest 行为测试，迁移真实 API 时补 adapter 和失败状态测试。
- 后端补单元测试、接口测试、订单状态机测试、支付回调幂等测试。
- 移动端 SDK 能力用 mock adapter 做自动化测试，真实设备测试单独列验收。
- 关键链路需要端到端测试：货主下单、司机接单、运输、确认、支付、评价。
- 后台需要权限测试和审计日志测试。

每个阶段必须通过：

- TypeScript 类型检查。
- lint。
- 相关单元测试。

## 14. 主要风险与控制

- 范围失控：每阶段只做一个可闭环目标，不跨阶段偷塞大功能。
- 账务风险：支付、退款、分账必须后端状态机和流水驱动。
- 订单状态混乱：订单状态只能由后端事件推进。
- 地图成本和稳定性：位置上报要限频，轨迹冷热数据分层。
- 第三方锁定：SDK 接入必须包在 adapter 后面，页面不直接依赖供应商。
- 当前代码大文件风险：迁移 API 前继续拆页面，不把新能力塞进大文件。
- 测试断层：本地 mock 测试要保留，真实接口测试逐步补。

## 15. 第一批实施建议

先做阶段 0 和阶段 1，不直接开司机端、后台、支付。

第一批最小可交付：

1. 新增 `docs` 级总体架构、ERD 初稿和 OpenAPI 初稿。
2. 选择后端技术栈并创建后端工程。
3. 实现真实认证、token refresh 和用户资料接口。
4. 货主端认证从本地固定验证码切到真实 API adapter。
5. 保持当前本地货主端其他功能不退化。

这样能先把“本地演示”变成“真实平台地基”。地基没打就开支付和司机端，那就是东北冬天穿拖鞋出门，嘴硬归嘴硬，脚是真遭罪。
