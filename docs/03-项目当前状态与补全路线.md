移动端 `platformOrderApi.listOrders()` 会先裁剪 `keyword` 两端空白，裁剪后为空的关键词不会发给后端；裁剪后超过 100 字符会在请求前抛出 `PLATFORM_ORDER_LIST_QUERY_INVALID`，避免明显非法的搜索条件继续打接口。

移动端 `platformOrderApi.listOrders()` 的查询参数本身运行时只接受对象；`null`、字符串、数字或数组都会在请求前抛出 `PLATFORM_ORDER_LIST_QUERY_INVALID`，不会被静默当成空查询或炸成原生 `TypeError`。

移动端 `platformOrderApi.listOrders()` 的 `keyword` 运行时只接受字符串，`null`、数字或对象等非字符串值都会在请求前抛出 `PLATFORM_ORDER_LIST_QUERY_INVALID`，不会被静默当成无关键词查询或炸成原生 `TypeError`。

移动端 `platformOrderApi.listOrders()` 会在发送 `statuses` 前按原顺序去重，和后端逗号集合归一化口径保持一致，避免重复状态把列表查询 URL 撑得又臭又长。

移动端 `platformOrderApi.listOrders()` 的 `createdFromIso/createdToIso` 运行时只接受字符串，会先裁剪两端空白，裁剪后为空的时间字段不会发给后端；`null`、数字、对象、不可解析时间和反向/空时间范围都会在请求前抛出 `PLATFORM_ORDER_LIST_QUERY_INVALID`。

移动端 `platformOrderApi.listOrders()` 的单状态 `status` 只要传入就必须是后端订单状态枚举，空字符串、`null` 或未知状态都会在请求前抛出 `PLATFORM_ORDER_LIST_QUERY_INVALID`，不会被静默当成无状态查询。

移动端 `platformOrderApi.listOrders()` 的状态集合 `statuses` 运行时只接受数组，字符串、`null` 或对象等非数组值都会在请求前抛出 `PLATFORM_ORDER_LIST_QUERY_INVALID`，避免集合筛选炸成原生 `TypeError` 或被错误拆成字符集合。

移动端 `platformOrderApi` 的订单详情、编辑、取消、确认送达、状态推进、异常上报、改单申请和评价接口会先裁剪 `orderId` 两端空白，裁剪后为空或运行时非字符串会在请求前抛出 `PLATFORM_ORDER_ID_INVALID`，不会把空路径、`null`、数字或对象拼进订单 URL。

移动端 `platformOrderApi.createOrder()` 和 `platformOrderApi.updateOrder()` 会在请求前按后端发单 schema 归一化订单主体：必填字符串裁剪后不能为空，空白的可选裁剪字段不发送，备注/描述长度、图片数量、手机号、装货时间、布尔字段、`pricingMode/paymentMethod`、固定价金额、议价订单夹带金额、优惠券字段成套性和实付金额关系都会先校验；非法请求会抛出 `PLATFORM_ORDER_REQUEST_INVALID`，不会把脏发单或编辑请求打到后端。

移动端 `platformProfileApi.saveAddressBook()` 会在请求前按后端地址簿 schema 归一化保存请求：地址和联系人数组必须存在且分别不超过 20/50 条，地址/联系人必填字符串会裁剪并校验长度，空白标签/备注和空白更新时间基线不会发送，联系人手机号必须符合大陆手机号格式，`clientUpdatedAtIso/baseUpdatedAtIso` 必须是可解析时间；非法请求会抛出 `PLATFORM_PROFILE_ADDRESS_BOOK_REQUEST_INVALID`，不会把脏地址簿快照打到后端。

移动端 `platformFrequentRoutesApi.saveFrequentRoutes()` 会在请求前按后端常用路线 schema 归一化保存请求：`routes` 必须是数组且最多 20 条，路线 ID、名称、装货地、卸货地和最近使用文案会裁剪并校验长度，空白 `lastUsedIso/clientUpdatedAtIso/baseUpdatedAtIso` 不发送，传入的时间字段必须可解析；非法请求会抛出 `PLATFORM_FREQUENT_ROUTES_REQUEST_INVALID`，不会把脏常用路线快照打到后端。

移动端 `platformOrderDraftApi.saveDraft()` 会在请求前按后端草稿 schema 归一化保存请求：请求体和 `draftSnapshot` 必须是普通对象，空白 `clientUpdatedAtIso/baseUpdatedAtIso` 不发送，传入的时间字段必须可解析；非法请求会抛出 `PLATFORM_ORDER_DRAFT_REQUEST_INVALID`，不会把脏草稿快照打到后端。

移动端 `platformAuthApi.refresh()` 和 `platformAuthApi.logout()` 会在请求前裁剪 `refreshToken/deviceId` 两端空白，只接受后端当前 opaque token 形态 `refresh.<UUID>`；非法 token session 请求会抛出 `PLATFORM_AUTH_TOKEN_SESSION_REQUEST_INVALID`，不会把旧语义 refresh token、access token、空设备 ID 或非对象请求打到后端。

### 3.1 发单流程

当前已打通本地发单闭环：

- 货物类型可选。
- 货物重量、体积、数量、描述可填写，货物描述最多 200 字。
- 货物重量必须包含有效数字，并满足 0.1 到 50 吨本地发布范围；支持 kg/公斤/千克文本按吨换算。
- 货物体积如填写需在 0.1 到 100 立方米之间。
- 货物数量必须包含有效正数，纯文本会拦截发布。
- 装货/卸货地址、50 字备注、联系人、电话可填写，并支持从个人中心当前本地常用地址建议一键回填地址、联系人和手机号，也支持从个人中心当前本地常用联系人建议一键回填装卸联系人和手机号。
- 装货地址和卸货地址相同时会拦截发布。
- 车型、车长、尾板和篷布要求可选择，装货时间需包含明确日期/相对日期和时间，并满足当前时间 2 小时后、7 天内、半小时刻度的本地发布规则。
- 期望送达时间支持填写明确时间或选择“尽快送达”，会进入确认页、订单详情和本地持久化订单状态；乱填文本会拦截发布。
- 支持装卸协助、保价运输、防震包装等增值服务本地选择；装卸协助可选择 1-5 人，保价运输需填写有效货值，并在确认页和订单详情展示。
- 支持一口价和议价模式；一口价需在 1 到 50000 元之间，议价订单可不填一口价发布，并在详情报价列表展示等待报价空状态。
- 支持货到付款和在线支付的本地选择，确认页和订单详情会展示支付方式；在线支付只提示本地不扣款，真实微信/支付宝待接入。
- 固定价订单可选择个人中心当前本地可用优惠券，按本地“满 X 减 Y”文案计算预览优惠金额和实付金额，并在订单详情和消费记录里展示原价、优惠券、优惠金额和实付金额；发布后会把该券在个人中心本地标记为已使用并记录订单号；编辑已带券待接单订单时会通过订单本地 `couponId` 保留当前券的预览和保存结果，若编辑时改为不使用优惠券，会把原本地券释放回“可使用”状态并记录取消使用的订单号；真实券核销、锁券、后端计价和支付抵扣仍未接入。
- 支持本地货物图片凭证 1-6 张计数占位、本地凭证清单和移除最新凭证，并在订单详情展示数量；超过 6 张会提示本地上限。
- 预览确认后会生成本地订单。
- 最终确认发布后跳转到订单详情。
- 未完成草稿会写入本地草稿存储适配层，App 重建后再次发单可恢复；适配层会记录版本和保存时间，并在读取到无效或超过 24 小时的快照时自动清理。

### 3.4 认证流程
